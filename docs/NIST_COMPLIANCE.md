# NIST SP 800-53 Rev 5 Compliance Overview

This document describes how **GovCon AI Scanner** maps to NIST SP 800-53 Rev 5 security controls and references the NIST SP 800-63B Digital Identity Guidelines.

---

## Access Control (AC)

### AC-2 — Account Management
- User accounts are stored in MongoDB with role, `isActive`, and audit timestamps (`createdAt`, `updatedAt`).
- Supported roles: `user`, `admin`, `capture`, `procurement`, `ops`, `exec`.
- Account creation, modification, and deactivation are each audited with structured `ACCOUNT_CREATED`, `ACCOUNT_UPDATED`, and `ACCOUNT_DEACTIVATED` events.
- MFA method enablement, disablement, and backup-code regeneration emit `ACCOUNT_UPDATED` events (tracked in `backend/routes/mfa.js`).
- Inactive accounts (`isActive: false`) are denied access at every protected endpoint.
- **Files:** `backend/models/User.js`, `backend/routes/admin.js`, `backend/routes/mfa.js`

### AC-3 — Access Enforcement
- All API routes are protected by `authenticateToken` (JWT verification).
- Role-based access is enforced via `requireRole(...roles)` middleware.
- `requireAdmin` middleware provides double enforcement on admin routes.
- **Files:** `backend/middleware/auth.js`, `backend/middleware/admin.js`

### AC-6 — Least Privilege
- Each role grants the minimum privileges needed for its function.
- Admin routes are further guarded by both `requireAuth` and `requireRole("admin")`.

### AC-7 — Unsuccessful Logon Attempts
- After **5 consecutive** failed login attempts (configurable via `LOGIN_MAX_FAILED_ATTEMPTS`), the account is locked for **15 minutes** (configurable via `LOGIN_LOCKOUT_MINUTES`).
- The failed attempt counter and lockout timestamp (`failedLoginAttempts`, `accountLockedUntil`) are stored per-user in the database.
- A successful login resets the counter.
- A password reset also clears the lockout.
- `ACCOUNT_LOCKED` audit events are emitted when a lockout is triggered.
- IP-level rate limiting via `express-rate-limit` provides an additional layer (AC-7 defense-in-depth).
- **Files:** `backend/models/User.js`, `backend/routes/auth.js`

### AC-11 / AC-12 — Session Termination
- Access tokens expire after **15 minutes** (short-lived).
- Refresh tokens expire after **7 days**.
- Logout invalidates the stored refresh-token hash.
- `LOGOUT` events are audited.
- **Files:** `backend/routes/auth.js`

### AC-17 — Remote Access
- CORS is restricted to an explicit allow-list (`ALLOWED_ORIGINS` env var).
- HTTP Strict Transport Security (HSTS) is enforced via Helmet with `maxAge=31536000` (1 year), `includeSubDomains`, and `preload`.
- **Files:** `server.js`

## Audit and Accountability (AU)

### AU-2 — Event Logging
The following events are captured in structured JSON audit records:

| Event | NIST Control |
|-------|-------------|
| `LOGIN_SUCCESS` / `LOGIN_FAILURE` | IA-2, IA-8 |
| `LOGOUT` | AC-12 |
| `TOKEN_REFRESH` | IA-11 |
| `MFA_SUCCESS` / `MFA_FAILURE` | IA-2(1) — emitted by `auth.js` (login verification) and `mfa.js` (setup verification) |
| `ACCOUNT_CREATED` | AC-2 |
| `ACCOUNT_UPDATED` | AC-2 |
| `ACCOUNT_DEACTIVATED` | AC-2 |
| `ACCOUNT_LOCKED` | AC-7 |
| `PASSWORD_RESET_REQUEST` | IA-5 |
| `PASSWORD_RESET_COMPLETE` | IA-5 |
| `PASSWORD_CHANGED` | IA-5 |
| `FILE_UPLOAD` | AU-2 |
| `ANALYSIS_RUN` | AU-2 |
| `ERP_TOKEN_CONNECTED` | IA-5(1) |
| `ERP_TOKEN_REFRESH` | IA-11 |
| `PERMISSION_DENIED` | AC-3 |
| `ADMIN_ACTION` | AC-6 |

### AU-3 — Content of Audit Records
Each audit record includes: `eventType`, `timestamp` (ISO-8601 UTC), `userId`, `email`, `ip`, `route`, `method`, `success`, `durationMs` (where applicable), and `details`.

Sensitive fields (`password`, `token`, `secret`, etc.) are automatically stripped before writing.

### AU-8 — Time Stamps
All timestamps use ISO-8601 UTC format (`new Date().toISOString()`).

### AU-9 — Protection of Audit Information
- Audit records are written to stdout for collection by the hosting platform (Render/Datadog).
- In development, records are also appended to `logs/audit.log`.
- Audit records are immutable once written (append-only log file).

### AU-11 — Audit Record Retention
- Production audit records are forwarded to stdout and should be captured by a log aggregation service (Datadog, Splunk, AWS CloudWatch, etc.) that enforces your organisation's retention policy.
- Recommended minimum retention: **90 days** (online) + **1 year** (offline/archive).

### AU-12 — Audit Record Generation
Audit records are generated by the centralised `audit()` function in `backend/services/auditLogger.js`, which is called from all authentication, authorisation, and account-management routes.

## Identification and Authentication (IA)

### IA-2 — Multi-Factor Authentication
- TOTP (Time-based One-Time Password) via an authenticator app is **mandatory** for all users.
- Users must complete TOTP setup before they can access any protected resource.
- Backup codes are available as a fallback (single-use, hashed at rest).
- All MFA success and failure events are audited with `MFA_SUCCESS` / `MFA_FAILURE` records across both the login flow (`/api/auth/verify-mfa-login`) and the setup flow (`/api/mfa/verify-totp-setup`, `/api/mfa/verify-setup`).
- **Files:** `backend/routes/auth.js`, `backend/routes/mfa.js`, `backend/services/totpService.js`

### IA-5 — Authenticator Management
- **Password policy (NIST SP 800-63B §5.1.1):**
  - Minimum length: **8 characters**
  - Maximum length: **128 characters** (prevents bcrypt DoS)
  - Checked against a list of known commonly-used passwords
  - No arbitrary complexity requirements (compliant with NIST 800-63B)
- Passwords are hashed with **bcrypt** (cost factor 12) before storage.
- TOTP secrets are encrypted at rest with **AES-256-GCM**.
- SMS phone numbers are encrypted at rest with **AES-256-GCM** (`MFA_ENCRYPTION_KEY`).
- Password reset tokens are stored as **SHA-256 hashes** (never in plaintext).
- Authenticated users can change their password via `POST /api/auth/change-password`.
- **Files:** `backend/models/User.js`, `backend/routes/auth.js`

### IA-8 — Non-Organisational Users
- All users are identified by a unique, lowercase email address.
- Duplicate email registration is rejected at the application layer.

### IA-11 — Re-Authentication
- ERP tokens are capped at a maximum age of **8 hours** (configurable via `ERP_MAX_TOKEN_AGE_HOURS`), regardless of ERP-issued TTL.
- Access token TTL is **15 minutes**, after which the client must use the refresh token.
- **Files:** `backend/models/ErpConfig.js`, `backend/routes/auth.js`

## System and Communications Protection (SC)

### SC-5 — Denial-of-Service Protection
All authentication and write endpoints are rate-limited using `express-rate-limit`:

| Endpoint group | Window | Max requests |
|----------------|--------|-------------|
| Login | 15 min | 10 |
| Registration | 15 min | 10 |
| MFA verification | 15 min | 5 |
| Password reset | 15 min | 5 |
| ERP authentication | 15 min | 10 |

### SC-8 — Transmission Confidentiality
- HTTP Strict Transport Security (HSTS) is enforced: `max-age=31536000; includeSubDomains; preload`.
- TLS termination is handled by the hosting platform (Render); the app should never be exposed on plain HTTP in production.

### SC-12 / SC-13 — Cryptographic Key Management
- **AES-256-GCM** (NIST FIPS 197 / SP 800-38D) is used for all symmetric encryption.
- A key version field (`encKeyVersion`) supports future key rotation.
- Encryption keys are loaded from environment variables, never hard-coded.
- **Files:** `backend/models/ErpConfig.js`, `backend/routes/mfa.js`, `backend/services/totpService.js`

### SC-23 — Session Authenticity
- JWT tokens are signed with HS256 (HMAC-SHA256) using a 64-byte random secret.
- Refresh tokens are stored as SHA-256 hashes; comparison uses `crypto.timingSafeEqual`.
- The `X-Frame-Options: SAMEORIGIN` header prevents clickjacking.

### SC-28 — Protection of Information at Rest
- ERP access tokens: encrypted with AES-256-GCM, never stored in plaintext.
- TOTP secrets: encrypted with AES-256-GCM.
- SMS phone numbers: encrypted with AES-256-GCM.
- Passwords: one-way bcrypt hash (cost 12).

## System and Information Integrity (SI)

### SI-10 — Information Input Validation
- NAICS codes are validated against a strict regex (`/^\d{2,6}$/`).
- User search queries in admin routes are regex-escaped to prevent ReDoS.
- Sort fields in admin routes are validated against an allow-list.
- Password input is validated for length and against a common-password list before any DB or crypto operation.
- JSON body size is limited to **4 MB** (prevents large-payload attacks).

## Configuration Management (CM)

### CM-6 — Configuration Settings
Key security-relevant configuration variables and their defaults:

| Variable | Default | Purpose |
|----------|---------|---------|
| `JWT_SECRET` | *(required)* | JWT signing secret |
| `JWT_REFRESH_SECRET` | `JWT_SECRET` | Refresh token signing secret |
| `LOGIN_MAX_FAILED_ATTEMPTS` | `5` | AC-7 lockout threshold |
| `LOGIN_LOCKOUT_MINUTES` | `15` | AC-7 lockout duration |
| `ERP_MAX_TOKEN_AGE_HOURS` | `8` | IA-11 max ERP token age |
| `ERP_MAX_FAILED_AUTH_ATTEMPTS` | `5` | AC-7 ERP lockout threshold |
| `ERP_AUTH_LOCKOUT_MINUTES` | `15` | AC-7 ERP lockout duration |
| `ERP_ENCRYPTION_KEY` | *(required)* | 32-byte hex AES-256 key |
| `MFA_ENCRYPTION_KEY` | `ERP_ENCRYPTION_KEY` | 32-byte hex AES-256 key for MFA data |
| `MFA_OTP_EXPIRY_MINUTES` | `5` | OTP validity window |
| `ALLOWED_ORIGINS` | localhost ports | CORS allow-list |

All secrets must be loaded from environment variables and must **never** be committed to source control.
